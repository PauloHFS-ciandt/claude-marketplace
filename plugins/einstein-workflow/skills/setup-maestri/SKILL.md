# /setup-maestri — Maestri Workspace Generator

You create a fully-wired Maestri workspace using the Maestri CLI. Terminals, roles, connections, and notes appear directly on the canvas — the user sees the result immediately.

**CLI resolution:** The Maestri CLI may be available as `maestri` in PATH or via the `$MAESTRI_CLI` environment variable (set automatically when the terminal is on a Maestri canvas). Throughout this skill, `maestri` means: use `$MAESTRI_CLI` if the env var is set, otherwise fall back to `maestri`.

## Step 0: Prerequisites

Before anything, verify BOTH conditions:

1. **CLAUDE.md exists** at the project root (generated by `/setup-project`). If not, tell the user to run `/setup-project` first and STOP.

2. **Maestri is running and connected** — try both CLI paths:
   ```bash
   $MAESTRI_CLI list 2>/dev/null || maestri list 2>/dev/null
   ```
   If BOTH fail, tell the user:
   > "Open the Maestri app and make sure this terminal is on the Maestri canvas before running /setup-maestri."
   
   Then STOP.
   
   **Remember which command worked** and use it for all subsequent Maestri commands in this skill.

3. **Check for existing terminals** — parse the `maestri list` output. If terminals named "Tech Lead", "Backend", "Frontend", "Mobile", or "AppSec" already exist, ask the user:
   > "I found existing terminals: {names}. Do you want to recreate them (dismiss + re-recruit) or keep the existing ones?"
   
   If recreating, dismiss them first:
   ```bash
   maestri dismiss "Tech Lead"
   maestri dismiss "Backend"
   ```
   (one command per terminal that needs replacing)

## Step 1: Read Project Config

Read `CLAUDE.md` from the project root. Extract and remember:
- **Project name** (the main heading or the first descriptive name)
- **Sub-projects** — each with: name, relative path, stack/technology description
- **Security gate** — whether AppSec review is mentioned as required

If `.claude/WORKFLOW.md` exists, read it too for additional context.

## Step 2: Ask Which Terminals

Use `AskUserQuestion` with multiSelect:

**Question:** "Which terminals do you want in your Maestri workspace?"

**Options:**
- Tech Lead (Recommended) — Orchestrates all work, delegates to specialists
- Backend Engineer — Backend implementation tasks
- Frontend Engineer — Frontend/CMS implementation tasks
- Mobile Engineer — Mobile app implementation tasks
- AppSec Engineer — Security review gate before PRs
- Shell — General-purpose shell terminal

Default recommendation: Tech Lead + one specialist per sub-project found in CLAUDE.md + AppSec.

## Step 3: Create Roles

For each selected terminal (except Shell), create a Maestri role.

### 3a. Build role prompts

For each terminal, build a prompt by combining:
1. The **full content** of the agent definition from `${CLAUDE_PLUGIN_ROOT}/agents/{agent-file}.md` (read the entire file including YAML frontmatter)
2. A **Project Context** section built from CLAUDE.md

**Agent file mapping:**

| Terminal | Agent File | Context Scope |
|----------|-----------|---------------|
| Tech Lead | `tech-lead.md` | Full CLAUDE.md + terminal topology table |
| Backend Engineer | `backend-engineer.md` | Only the backend sub-project section from CLAUDE.md |
| Frontend Engineer | `frontend-engineer.md` | Only the frontend sub-project section from CLAUDE.md |
| Mobile Engineer | `mobile-engineer.md` | Only the mobile sub-project section from CLAUDE.md |
| AppSec Engineer | `security-reviewer.md` | Full CLAUDE.md (needs cross-project visibility) |

**For Tech Lead**, append this after the agent content:

```markdown
## Project Context

{paste full CLAUDE.md content here}

## Maestri Terminals

| Terminal | Role |
|----------|------|
| Tech Lead | Orchestration, architecture, PR workflow |
| Backend | {backend sub-project description} |
| Frontend | {frontend sub-project description} |
| Mobile | {mobile sub-project description} |
| AppSec | Security review gate |

## Delegation Protocol

You delegate ALL implementation via `maestri ask`:
- `maestri ask "Backend" "implement the endpoint per the spec"`
- `maestri ask "AppSec" "review the current diff for security"`

You NEVER implement directly. You orchestrate, review, and approve.
```

**For specialists** (Backend, Frontend, Mobile), append:

```markdown
## Project Context

{paste ONLY the relevant sub-project section from CLAUDE.md}

Working directory: {sub-project relative path}
```

**For AppSec**, append:

```markdown
## Project Context

{paste full CLAUDE.md content here}
```

### 3b. Write prompt to temp file and create role

The prompt content contains quotes, newlines, and special characters that break shell argument passing. For each role:

1. Write the built prompt to a temp file:
   ```bash
   cat > /tmp/maestri-role-techlead.md << 'ROLE_EOF'
   {the full prompt content}
   ROLE_EOF
   ```

2. Create the role:
   ```bash
   maestri role create "{ProjectName} - Tech Lead" "$(cat /tmp/maestri-role-techlead.md)"
   ```

3. Clean up:
   ```bash
   rm /tmp/maestri-role-techlead.md
   ```

Repeat for each terminal. Use descriptive temp file names: `maestri-role-backend.md`, `maestri-role-frontend.md`, etc.

## Step 4: Recruit Terminals

Create each terminal on the canvas. Run these **sequentially** with a 2-second pause between each:

```bash
maestri recruit "Tech Lead" --preset "Claude Code" --role "{ProjectName} - Tech Lead"
sleep 2
maestri recruit "Backend" --preset "Claude Code" --role "{ProjectName} - Backend Engineer"
sleep 2
maestri recruit "Frontend" --preset "Claude Code" --role "{ProjectName} - Frontend Engineer"
sleep 2
maestri recruit "Mobile" --preset "Claude Code" --role "{ProjectName} - Mobile Engineer"
sleep 2
maestri recruit "AppSec" --preset "Claude Code" --role "{ProjectName} - AppSec Engineer"
```

If Shell was selected:
```bash
sleep 2
maestri recruit "Shell" --preset "Claude Code"
```

Only recruit terminals that the user selected in Step 2.

**If `maestri recruit` fails:** check `maestri preset list` to verify "Claude Code" exists. If not, tell the user to configure Claude Code in Maestri's settings.

## Step 5: Connect Terminals

Wire all specialists to the Tech Lead (creates visible "ropes" on the canvas):

```bash
maestri connect "Backend" "Tech Lead"
maestri connect "Frontend" "Tech Lead"
maestri connect "Mobile" "Tech Lead"
maestri connect "AppSec" "Tech Lead"
```

Only connect terminals that were actually recruited in Step 4. Shell is NOT connected.

**If `maestri connect` fails:** wait 3 seconds and retry once — terminals may still be initializing.

## Step 6: Create Shared Note

Create a project overview sticky note. Use `$'...'` syntax for real newlines:

```bash
maestri note create $'# {ProjectName}\n\n{2-3 sentence project summary}\n\n## Sub-Projects\n| Name | Path | Stack |\n|------|------|-------|\n| {name} | {path} | {stack} |'
```

The note is auto-connected to the current terminal and auto-named from the heading in kebab-case (e.g., `# My App` becomes `my-app`).

Then connect it to the Tech Lead terminal too:
```bash
maestri connect "Tech Lead" "{kebab-case-project-name}"
```

## Step 7: Update WORKFLOW.md

Append or replace the `## Maestri Workspace` section in `.claude/WORKFLOW.md`:

```markdown
## Maestri Workspace

| Terminal | Role | Scope |
|----------|------|-------|
| Tech Lead | Orchestration | Full project |
| Backend | Implementation | {backend path} |
| Frontend | Implementation | {frontend path} |
| Mobile | Implementation | {mobile path} |
| AppSec | Security gate | Full project |

### Delegation Protocol

The Tech Lead delegates via `maestri ask`, NOT via the Agent tool.
All specialists report back to Tech Lead after completing tasks.
AppSec review is MANDATORY before any PR is created.
```

## Step 8: Verify & Report

Run `maestri list` and confirm all terminals appear.

Then print the topology:

```
{ProjectName} Maestri Workspace

├── Tech Lead ←──┐
│   ├── → Backend
│   ├── → Frontend
│   ├── → Mobile
│   └── → AppSec (security gate)
├── Shell (standalone)
└── Note: {ProjectName} overview

Ready! Talk to "Tech Lead" to start working.
Example: maestri ask "Tech Lead" "plan the next sprint"
```

## Re-running the Skill

This is handled in Step 0. If terminals exist, the user chooses to recreate or keep them. When recreating:
1. `maestri dismiss` each old terminal
2. Roles are NOT deleted (Maestri accumulates them — duplicates with the same name are fine, the latest is used)
3. Proceed from Step 3

## Important Notes

- **Terminal names must be short and unique.** Use "Tech Lead", "Backend", "Frontend", "Mobile", "AppSec", "Shell" — not the full project-qualified names.
- **Role names ARE project-qualified** (e.g., "MyApp - Tech Lead") to avoid conflicts across workspaces.
- **The agent files in `${CLAUDE_PLUGIN_ROOT}/agents/`** are the source of truth for methodology. The roles bake a snapshot of them + project context. If agent definitions change, re-run `/setup-maestri`.
