# /setup-maestri — Maestri Workspace Generator

You generate a Maestri workspace configuration so the user can orchestrate specialist agents from dedicated terminals.

## Prerequisites

- The project must have a `CLAUDE.md` at the root (generated by `/setup-project`)
- The project must have `.claude/WORKFLOW.md` (generated by `/setup-project`)
- Maestri must be installed on the user's machine

## Wizard Flow

### Step 1: Read Existing Config

Read `CLAUDE.md` and `.claude/WORKFLOW.md` to understand:
- Project name
- Sub-projects (name, path, stack)
- Whether security gate is enabled
- Team topology

### Step 2: Confirm Terminals

Ask the user using `AskUserQuestion`:

1. "Which terminals do you want in your Maestri workspace?" (multiSelect)
   - Tech Lead (recommended — always include)
   - Backend Engineer
   - Frontend Engineer
   - Mobile Engineer
   - AppSec Engineer (for security review gate)
   - Podman/Docker Compose (infrastructure terminal)
   - Shell (for claude-mem or general shell tasks)

### Step 3: Generate .maestri/ Structure

For each selected terminal, create:

```
.maestri/
  roles/
    {uuid}/
      AGENTS.md    — Role system prompt with project context baked in
      CLAUDE.md    — Working directory and basic context
```

Generate a UUID for each role (use `uuidgen` command or a random hex string).

#### AGENTS.md Template (per role)

The AGENTS.md for each Maestri role should contain the SAME content as the corresponding plugin agent (from `~/.claude/agents/`), BUT with project context from CLAUDE.md baked directly into the prompt. This is because Maestri terminals may not inherit global agent definitions.

**Tech Lead AGENTS.md:**
```markdown
# {project_name} — Tech Lead

You are the Staff Engineer and Tech Lead for **{project_name}**.

{Paste the full tech-lead methodology from the plugin agent}

## Project Context
{Paste the relevant sections from CLAUDE.md: overview, structure, team, Azure DevOps, guidelines}

## Maestri Terminals
| Terminal | Role | Working Directory |
|---|---|---|
{For each terminal created, list name, role, and path}
```

**Specialist AGENTS.md (Backend/Frontend/Mobile):**
```markdown
# {project_name} — {Role}

You are the {Role} for **{project_name}**.

{Paste the full role methodology from the plugin agent}

## Project Context
{Paste ONLY the relevant sub-project section from CLAUDE.md}
```

**AppSec AGENTS.md:**
```markdown
# {project_name} — AppSec Engineer

You are the Application Security Engineer for **{project_name}**.

Your job is to review code changes for security vulnerabilities before PRs are created.

## Security Review Checklist

For every review, check:

### Backend Security
- [ ] Input validation on all user inputs
- [ ] SQL injection prevention (parameterized queries)
- [ ] Authentication/authorization on all endpoints
- [ ] No secrets in code or logs
- [ ] Rate limiting on sensitive endpoints
- [ ] CORS properly configured
- [ ] Headers: CSP, X-Frame-Options, etc.
- [ ] Dependency vulnerabilities (npm audit)

### Frontend/Mobile Security
- [ ] XSS prevention (no dangerouslySetInnerHTML without sanitization)
- [ ] CSRF protection
- [ ] Sensitive data not in localStorage/AsyncStorage
- [ ] Auth tokens properly managed
- [ ] No exposed API keys in client code

### API Security
- [ ] Authentication required on all non-public endpoints
- [ ] Authorization checked (user can only access own resources)
- [ ] Input validation and sanitization
- [ ] Error messages don't leak internal details
- [ ] File upload validation (type, size)

## Output Format

After review, respond with:
1. **PASS** or **FAIL** (with blocking issues)
2. List of findings: severity (CRITICAL/HIGH/MEDIUM/LOW), description, file, recommendation
3. Summary recommendation
```

#### CLAUDE.md Template (per role)

Each role's CLAUDE.md sets the working directory context:

```markdown
# {role_name}

Working directory: `{working_directory}`
Project: {project_name}

@../../CLAUDE.md
```

### Step 4: Update .claude/WORKFLOW.md

Update the existing `.claude/WORKFLOW.md` to include the Maestri terminal mapping:

```markdown
## Maestri Workspace

| Terminal | Role UUID | Working Directory |
|---|---|---|
| Tech Lead | {uuid} | `{root}/` |
| Backend | {uuid} | `{backend_path}/` |
...

## Delegation Protocol

The Tech Lead delegates via Maestri (`maestri ask`), NOT via the Agent tool.
Specialists are stateless — they receive full context via delegation messages.
```

### Step 5: Print Summary

Show the user:

1. The complete terminal topology as a tree:
```
{project_name} Maestri Workspace
├── Tech Lead ({root}/)
│   ├── delegates to → Backend ({backend_path}/)
│   ├── delegates to → Frontend ({frontend_path}/)
│   ├── delegates to → Mobile ({mobile_path}/)
│   └── delegates to → AppSec ({root}/)
├── Podman Compose ({infra_path}/) [if selected]
└── Shell ({root}/) [if selected]
```

2. How to use:
   - "Open Maestri and load this workspace"
   - "Talk to the Tech Lead terminal — it will delegate to specialists"
   - "The AppSec terminal is the mandatory gate before PRs"

3. Sync reminder:
   - "Each agent has TWO copies: the plugin version (dynamic) and the Maestri version (baked-in)"
   - "If you change CLAUDE.md, re-run `/setup-maestri` to re-sync Maestri roles"

## Important Notes

- Maestri role UUIDs are generated fresh each time. If you re-run this skill, it creates NEW roles (old ones are not deleted automatically).
- The Tech Lead's AGENTS.md includes the full team topology so it knows who to delegate to.
- Specialist AGENTS.md files only include their own sub-project context (not the full monorepo) to keep context focused.
